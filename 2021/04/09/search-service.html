暗藏的数据：spu的id，sku的id

1.2.3.最终的数据结构

我们创建一个类，封装要保存到索引库的数据，并设置映射属性：

public class Goods {

    @Field(type = FieldType.Keyword, index = false)

    private Long cid2;// 2级分类id
    private Long cid3;// 3级分类id
    private Date createTime;// 创建时间
    private Set<Long> price;// 价格
    @Field(type = FieldType.Keyword, index = false)
    private String skus;// sku的信息集合存成json结构 展示
    private Map<String, Object> specs;// 可搜索的规格参数，key是参数名，值是参数值
    //get,set略
}

JAVA

一些特殊字段解释：

  • all：用来进行全文检索的字段，里面包含标题、商品分类信息

  • price：价格数组，是所有sku的价格集合。方便根据价格进行筛选过滤

  • skus：用于页面展示的sku信息，不索引，不搜索。包含skuId、image、price、title字段

  • specs：所有规格参数的集合。key是参数名，值是参数值。

    例如：我们在specs中存储 内存：4G,6G，颜色为红色，转为json就是：

    {
    "specs":{
        "内存":[4G,6G],
        "颜色":"红色"
    }
}

    JSON

    当存储到索引库时，elasticsearch会处理为两个字段：

    • specs.内存：[4G,6G]
    • specs.颜色：红色

    另外， 对于字符串类型，还会额外存储一个字段，这个字段不会分词，用作聚合。

    • specs.颜色.keyword：红色

    specs不可以设置成对象，如果这设置成对象，那么每类商品的规格参数是不一样的，那这个对象要涵盖好多的属性在一起，这里用map，因为如果是一个类或者一个对象它的key（属性名）必须是确定的，值是不确定的，但是map的key和value都可以是不确定的。

    // 但在这里接口的返回值可以写成ResponseEntity<List<Category>>也可以直接写成List<Category>
    // 因为ResponseEntity只是告诉mvc 结果写到响应体 返回的结果还是List<Category>

    List<Category> queryCategoryByIds(@RequestParam("ids")List<Long> ids);
}

JAVA

以上的方法直接从商品微服务中拷贝而来，差别就是没有方法的具体实现。大家觉得这样有没有问题？

其实如果没有特殊的配置的话，FeignClient的代码是遵循SpringMVC的风格的，因此与商品微服务的Controller完全一致。这样就存在一定的问题：

  • 代码冗余。尽管不用写实现，只是写接口，但服务调用方要写与服务controller一致的代码，有几个消费者就要写几次。
  • 增加开发成本。调用方还得清楚知道接口的路径，才能编写正确的FeignClient。
  • 如果生产者（服务提供方）更改了接口的话，那么所有的消费者（服务调用方）都需要更改。

解决方案

一种比较友好的实践是这样的：

  • 我们的服务提供方不仅提供实体类，还提供Api接口声明。
  • 调用方不用自己编写接口方法声明，直接继承提供方给的Api接口即可。

其实说白了就是服务的提供方写好controller之后，将controller中的方法（不带实现）copy到Api接口声明中一份即可，需要注意的是方法上的接口路径，因为有的controller的类上可能有一段路径，copy的时候要拼接上去。

/**
 * @Classname SearchServiceImpl
 * @Description TODO
 * @Date 2021/2/27 11:05
 * @Created by ChenSiYu
 */
@Service
public class SearchServiceImpl implements SearchService {

    @Autowired
    private CategoryClient categoryClient;
    @Autowired
    private BrandClient brandClient;
    @Autowired
    private GoodsClient goodsClient;
    @Autowired
    private SpecificationClient specificationClient;

    @Override
    public Goods buildGoods(SpuVo spu) {
        // 查询分类
        List<Category> categoryList = categoryClient.queryCategoryByIds(Arrays.asList(spu.getCid1(), spu.getCid2(), spu.getCid3()));
        if (CollectionUtils.isEmpty(categoryList)) {
            throw new LyException(ExceptionEnum.CATEGORY_NOT_FOUND);
        }
        List<String> cNames = categoryList.stream().map(Category::getName).collect(Collectors.toList());
        // 查询品牌
        Brand brand = brandClient.queryBrandById(spu.getBrandId());
        if (brand == null) {
            throw new LyException(ExceptionEnum.BRAND_NOT_FOUND);
        }
        String bName = brand.getName();
        // 标题
        String title = spu.getTitle();
        // 拼接搜索字段
        String all = StringUtils.join(cNames, " ") + bName + title;

        // 查询sku
        List<Sku> skuList = goodsClient.findSkuList(spu.getId());
        if (CollectionUtils.isEmpty(skuList)) {
            throw new LyException(ExceptionEnum.GOODS_NOT_FOUND);
        }
        // 所有sku的价格集合
        Set<Long> prices = skuList.stream().map(Sku::getPrice).collect(Collectors.toSet());

        // sku集合的json格式
        // sku中的数据很多 我们只需要id,title,images,price就够了
        // 所以要对sku进行处理 可以自己封装一个类 将需要的属性放里就可
        // 也可以定义一个map 因为map也可以存k-v结构 只不过对象的k固定 map的k可任意
        List<Map<String, Object>> skus = new ArrayList<>();
        for (Sku sku : skuList) {
            Map<String, Object> map = new HashMap<>();
            map.put("id", sku.getId());
            map.put("title", sku.getTitle());
            map.put("price", sku.getPrice());
            // 截取sku.getImages()里面第一个,之前的图片
            map.put("images", StringUtils.substringBefore(sku.getImages(), ","));

    private String chooseSegment(String value, SpecParam p) {
        double val = NumberUtils.toDouble(value);
        String result = "其它";
        // 保存数值段
        for (String segment : p.getSegments().split(",")) {
            String[] segs = segment.split("-");
            // 获取数值范围
            double begin = NumberUtils.toDouble(segs[0]);
            double end = Double.MAX_VALUE;
            if (segs.length == 2) {
                end = NumberUtils.toDouble(segs[1]);
            }
            // 判断是否在范围内
            if (val >= begin && val < end) {
                if (segs.length == 1) {
                    result = segs[0] + p.getUnit() + "以上";
                } else if (begin == 0) {
                    result = segs[1] + p.getUnit() + "以下";
                } else {
                    result = segment + p.getUnit();
                }
                break;
            }
        }
        return result;
    }

5.生成规格参数过滤

5.1.谋而后动

有四个问题需要先思考清楚：

  • 什么时候显示规格参数过滤？
  • 如何知道哪些规格需要过滤？
  • 要过滤的参数，其可选值是如何获取的？
  • 规格过滤的可选值，其数据格式怎样的？

什么情况下显示有关规格参数的过滤？

如果用户尚未选择商品分类，或者聚合得到的分类数大于1，那么就没必要进行规格参数的聚合。因为不同分类的商品，其规格是不同的。

因此，我们在后台需要对聚合得到的商品分类数量进行判断，如果等于1，我们才继续进行规格参数的聚合。

如何知道哪些规格需要过滤？

我们不能把数据库中的所有规格参数都拿来过滤。因为并不是所有的规格参数都可以用来过滤，参数的值是不确定的。值得庆幸的是，我们在设计规格参数时，已经标记了某些规格可搜索，某些不可搜索。

因此，一旦商品分类确定，我们就可以根据商品分类查询到其对应的规格，从而知道哪些规格可进行过滤。

要过滤的参数，其可选值是如何获取的？

虽然数据库中有所有的规格参数，但是不能把一切数据都用来供用户选择。与商品分类和品牌一样，应该是从用户搜索得到的结果中以规格参数聚合，得到商品的规格参数可选值。

     * @Description 构建Goods时对于区间的处理
     * @Param [value, p]
     * @Date 2021/3/2 14:14
     * @Author ChenSiYu
     */
    private String chooseSegment(String value, SpecParam p) {
        double val = NumberUtils.toDouble(value);
        String result = "其它";
        // 保存数值段
        for (String segment : p.getSegments().split(",")) {
            String[] segs = segment.split("-");
            // 获取数值范围
            double begin = NumberUtils.toDouble(segs[0]);
            double end = Double.MAX_VALUE;
            if (segs.length == 2) {
                end = NumberUtils.toDouble(segs[1]);
            }
            // 判断是否在范围内
            if (val >= begin && val < end) {
                if (segs.length == 1) {
                    result = segs[0] + p.getUnit() + "以上";
                } else if (begin == 0) {
                    result = segs[1] + p.getUnit() + "以下";
                } else {
                    result = segment + p.getUnit();
                }
                break;
            }
        }
        return result;
    }

拓展请求对象

我们需要在请求类：SearchRequest中添加属性，接收过滤属性。过滤属性都是键值对格式，但是key不确定，所以用一个map来接收即可。所以有的时候的业务场景，用map比创建一个对象更好，如果要存的值不多无所谓，但是像上面这种业务，存规格参数，每类商品有一套规格参数，有N类商品，每套里面又有N多规格参数，这种数据量很大的话，如果我们新建一个规格参数的对象，无法将这么多的规格参数全部囊括，因为对象的属性名定义后就是固定的，而Map的key却是灵活的。

private QueryBuilder buildBasicQueryWithFilter(SearchRequest request) {
    // 创建布尔查询 布尔查询分为must和filter
    BoolQueryBuilder queryBuilder = QueryBuilders.boolQuery();
    // 基本查询条件
    queryBuilder.must(QueryBuilders.matchQuery("all", request.getKey()).operator(Operator.AND));
    // 过滤条件
    Map<String, String> filter = request.getFilter();
    for (Map.Entry<String, String> entry : filter.entrySet()) {
        String key = entry.getKey();
        String value = entry.getValue();
        // 处理key
        // 商品分类和品牌只需传key key就是cid3或者brandId
        // 按照规格需要specs.key.keyword拼接 key就是规格参数名
        if (key != "cid3" && key != "brandId") {
            key = "specs." + key + ".keyword";
        }
        // keyword字符串类型，进行term查询 精确查询
        queryBuilder.filter(QueryBuilders.termQuery(key, value));
    }
    return queryBuilder;
}
