电商项目中商品的分析和数据库设计

前言：众所周知，电商网站品类复杂多样，比如京东、淘宝、拼多多等等，诸如这些全品类的电商网站，商品的种类繁多，每一件商品，其属性又有差别。那么设计如此大型的电商网站，我们该如何设计商品信息的展示呢？本文大部分以京东官网的截图为例说明思路。

一、SPU和SKU

一个全品类的电商网站，商品的种类繁多，每一件商品，其属性又有差别。为了更准确描述商品及细分差别，抽象出两个概念：SPU 和 SKU，了解一下：

  • SPU：Standard Product Unit （标准产品单位） ，一组具有共同特性的商品集。
  • SKU：Stock Keeping Unit（库存量单位），SPU商品集因具体特性不同而细分的每个商品。

以图为例来看：

  • 本页的华为 Mate10 就是一个商品集。（SPU）
  • 因为颜色、内存等不同，而细分出不同的Mate10，如亮黑色128G版。（SKU）

可以看出：

  • SPU是一个抽象的商品集概念，为了方便后台的管理。
  • SKU才是具体要销售的商品，每一个SKU的价格、库存可能会不一样，用户购买的是SKU而不是SPU。

二、思考并发现问题

弄清楚了SPU和SKU的概念区分，接下来我们一起思考一下该如何设计数据库表。

首先来看SPU，大家一起思考下SPU应该有哪些字段来描述？

似乎并不复杂，但是我们仔细思考一下，商品的规格字段你如何填写？

不同商品的规格不一定相同，数据库中要如何保存？难道要将全部商品的规格参数名全部列出来作为数据库的字段名？就这？

再看下SKU，大家觉得应该有什么字段？

综上我们发现：虽然不同商品的规格不同，但是同一分类下的商品，比如都是手机，其规格属性名是一致的，但是规格属性值不一样。也就是说，商品的规格参数应该是和分类绑定的。每一个分类都有统一的规格参数模板，但是同一分类下的不同商品其规格参数值可能不同。

因此：

  • 规格参数的名称（key）和值（value）应该分开来保存。
  • 一个分类对应一套规格参数模板（一对一），只有规格参数名（key），没有值。
  • 一个分类对应多个商品SPU（一对多）。
  • 一套规格模板对应多个商品SPU（一对多）。
  • 商品SPU保存规格参数的具体值。

1、SKU的特有属性

SPU中会有一些特殊属性，用来区分不同的SKU，我们称为SKU特有属性。如华为META10的颜色、内存属性。

不同种类的商品，一个手机，一个衣服，其SKU属性不相同。

同一种类的商品，比如都是衣服，SKU属性基本是一样的，都是颜色、尺码等。

这样说起来，似乎SKU的特有属性也是与分类相关的？事实上，仔细观察你会发现，SKU的特有属性是商品规格参数的一部分：

也就是说，我们没必要单独对SKU的特有属性进行设计，它可以看做是规格参数中的一部分。这样规格参数中的属性可以标记成两部分：

  • 所有sku共享的规格属性（称为全局属性）
  • 每个sku不同的规格属性（称为特有属性）

很奇怪是吧，只有两个字段。特别需要注意的是第二个字段：

  • specificatons：规格参数模板，json格式

为什么是一个json？我们看下规格参数的格式：

如果按照传统数据库设计，这里至少需要3张表：

  • tb_spec_group：代表组，与商品分类关联，一对多，一个分类下有多个组
  • tb_spec_param：属性名，与组关联，一对多，一个组内有多个规格属性名

这样程序的复杂度大大增加，但是提高了数据的复用性。后面说这种方式的实现。

我们的第一种解决方案是，采用Json来保存整个规格参数模板，不需要额外的表，一个字符串就够了。

Json结构分析：

整体：

整体

  • 因为规格参数分为很多组，所以json最外层是一个数组。
  • 数组中是对象类型，每个对象代表一个组的数据，对象的属性包括：
    • group：组的名称
    • params：该组的所有属性

params：

params

以主芯片这一组为例：

  • group：组名，这里是主芯片
  • params：该组的所有规格属性，因为不止一个，所以是一个数组。这里包含四个规格属性：CPU品牌，CPU型号，CPU频率，CPU核数。每个规格属性都是一个对象，包含以下信息：
    • k：属性名称
    • searchable：是否作为搜索字段，将来在搜索页面使用，boolean类型
    • global：是否是SPU全局属性，boolean类型。true为全局属性，false为SKU的特有属性
    • options：属性值的可选项，数组结构。起约束作用，不允许填写可选项以外的值，比如CPU核数，有人添10000核岂不是很扯淡
    • numerical：是否为数值，boolean类型，true则为数值，false则不是。为空也代表非数值
    • unit：单位，如：克，毫米。如果是数值类型，那么就需要有单位，否则可以不填。

上面的截图中所有属性都是全局属性，我们来看看内存，应该是特有属性：

总结下：

  • 规格参数分组，每组有多个规格参数
  • 规格参数的 k 代表规格参数名称，没有值，具体的SPU才能确定值
  • 规格参数会有不同的属性：是否可搜索，是否是全局、是否是数值，这些都用boolean值进行标记：
    • SPU下的多个SKU共享的规格参数称为全局属性，用global标记
    • SPU下的多个SKU特有的规格参数称为特有属性
    • 如果规格参数是数值类型，用numerical标记，并且指定单位unit
    • 如果规格参数可搜索，用searchable标记

2、数据库保存整个规格参数模板

值我们暂且不管，等到新增商品时，再来填写规格参数值即可，我们先思考规格参数名（key）该如何设计。

来看下商品规格的整体数据结构，我们先关注左边的key。

我们可以看到：

  • 规格数据首先要分组，组内再有不同的规格参数。
  • 不同的分类，其分组名称不同。
  • 不同的分类，组内规格参数也不同。

这样就意味着，有多少分类，就有多少分类下的分组，至少有数千数据，组内属性也是一样，数量更多。

如果按照传统设计，我们会以规格参数作为数据库的字段名，如品牌、型号等都是字段，那么就意味着表的字段会有无限多。如此设计的表称为横表。一条信息，描述商品的所有数据，称之为商品表。

显然我们不会这样做。

我们一条数据，只描述一条规格属性名，也就是将规格参数名作为字段的值，而非字段本身（字段名），这样的设计称为竖表设计。不过，按照上图的样式，我们还涉及到了分组，因此，规格参数名和规格组也要单独保存，都采用竖表设计。

所以我们有两张表：

  • 规格组：tb_spec_group
    • 一个商品分类下有多个规格组
  • 规格参数：tb_spec_param
    • 一个规格组下有多个规格参数

    @GetMapping("/params")

            @RequestParam(value = "gid", required = false) Long gid,
            @RequestParam(value = "cid", required = false) Long cid,
            @RequestParam(value = "searching", required = false) Boolean searching,
            @RequestParam(value = "generic", required = false) Boolean generic

        }
        return ResponseEntity.ok(list);
    }

    @Override
    public PageResult<SpuVo> querySpuByPageAndSort(Integer page, Integer rows, Boolean saleable, String key) {
        // 查询Spu
        // 分页,最多允许查100条
        PageHelper.startPage(page, Math.min(rows, 100));
        // 创建查询条件
        Example example = new Example(Spu.class);
        Example.Criteria criteria = example.createCriteria();
        // 是否过滤上下架
        if (saleable != null) {
            criteria.andEqualTo("saleable", saleable);
        }
        // 是否模糊查询 搜索字段过滤
        if (StringUtils.isNotBlank(key)) {
            criteria.andLike("title", "%" + key + "%");
        }
        // 默认排序
        example.setOrderByClause("last_update_time DESC");
        // 查询
        List<Spu> spus = this.spuMapper.selectByExample(example);
        // 判断
        if (CollectionUtils.isEmpty(spus)) {
            throw new LyException(ExceptionEnum.GOODS_NOT_FOUND);
        }

        // 将cname和bname加入 变成SpuVo的集合
        List<SpuVo> spuVos = loadCategoryAndBrandName(spus);

        // 解析最终的分页结果
        PageInfo<SpuVo> info = new PageInfo<>(spuVos);
        PageResult<SpuVo> result = new PageResult<>(info.getTotal(), spuVos);
        return result;
    }

    /**
     * @return java.util.List<com.leyou.item.vo.SpuVo>
     * @Description 加载分类名称和品牌名称
     * @Param [spus]
     * @Date 2021/2/23 16:38
     * @Author ChenSiYu
     */
    private List<SpuVo> loadCategoryAndBrandName(List<Spu> spus) {
        List<SpuVo> spuVos = new ArrayList<>(spus.size());
        for (Spu spu : spus) {
            // 把spu变为 spuVo
            SpuVo spuVo = new SpuVo();
            // 属性拷贝
            BeanUtils.copyProperties(spu, spuVo);
            // 处理分类名称
            List<Category> categoryList = this.categoryService.queryCategoryListByIds(Arrays.asList(spu.getCid1(), spu.getCid2(), spu.getCid3()));
//            List<String> names = categoryList.stream().map(category -> category.getName()).collect(Collectors.toList());
            // 先将category的集合的流通过map映射成cname的集合的流再将这个流转成新的集合也就是cname的集合
            List<String> names = categoryList.stream().map(Category::getName).collect(Collectors.toList());
            // 将分类名称拼接后存入 将集合拼成字符串后放入cname属性中

    @Override
    @Transactional
    public void updateGoods(SpuBo spuBo) {
        // 通过spu的id 查询以前的sku
        List<Sku> skus = this.querySkuListBySpuId(spuBo.getId());
        // 如果以前存在sku则删除
        if (!CollectionUtils.isEmpty(skus)) {
            // 将Sku的list集合变成流,再将Sku映射到sku的id属性,最后再将其变成sku的id属性的集合
            List<Long> ids = skus.stream().map(Sku::getId).collect(Collectors.toList());

            // 删除以前的库存
            // stock的主键是skuId对应Sku的id 要删除所有的stock 所以要根据sku的id的集合对应删除
            Example example = new Example(Stock.class);
            Example.Criteria criteria = example.createCriteria();
            criteria.andIn("skuId", ids);
            stockMapper.deleteByExample(example);

            // spu与sku是一对多 因此要增加或删除sku就要用spu的id
            // 增加或删除某个商品的sku的时候,这个商品所属商品集下的所有商品都要跟着变
            // 删除以前的sku
            Sku record = new Sku();
            record.setSpuId(spuBo.getId());
            skuMapper.delete(record);
        }

        // 新增sku和库存
        saveSkuAndStock(spuBo.getSkus(), spuBo.getId());

        // 更新spu
        spuBo.setLastUpdateTime(new Date());
        spuBo.setCreateTime(null);
        spuBo.setValid(null);
        spuBo.setSaleable(null);
        //updateByPrimaryKey对你注入的字段全部更新（不判断是否为Null）
        //updateByPrimaryKeySelective会对字段进行判断再更新(如果为Null就忽略更新)
        spuMapper.updateByPrimaryKeySelective(spuBo);

        // 更新spu详情
        spuDetailMapper.updateByPrimaryKeySelective(spuBo.getSpuDetail());

八、总结

现总结一下：

一种商品分类对应一套规格模板；

一套规格模板对应多个规格组；

一个规格组里面包含多个规格参数；

一套规格模板可以对应多个SPU；

一个SPU是一个商品集，里面包含商品的公有属性和所有特有属性的可选项；

一个SKU是一个商品，里面包含商品的公有属性和明确（选好）的特有属性；

我们这里的设计是将公有属性和所有特有属性的可选项存入SPU表，将特有属性存入SKU表；

一个SPU包含多个SKU；

为提高读SPU的效率，将商品的大属性（比如商品描述等）单独抽出到SpuDetail中；

库存本是特有属性，属于SKU的，但是由于库存需要频繁的读写，而SKU中的数据大部分只进行读操作，因此为了提高读SKU的效率，将库存参数单独分离出来，独立于SKU存在，Stock。

SPU数据可以修改，但是SKU数据无法修改，因为有可能之前存在的SKU现在已经不存在了，或者以前的SKU属性都不存在了。比如以前内存有4G，现在没了。因此这里直接删除以前的SKU，然后新增即可。

SPU与SKU是一对多，因此要增加或删除SKU就要用SPU的id，增加或删除某个商品的SKU的时候，这个商品所属商品集下的所有商品都要跟着变。

SpuDetail的主键spuId对应跟随Spu的主键id，Stock的主键skuId对应跟随Sku的主键id，操作SKU的时候要用SPU的id。
