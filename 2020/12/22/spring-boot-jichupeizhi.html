Spring Boot的基础配置

1. 容器的相关配置

众所周知，Spring Boot 的启动不同于普通的 SSM 项目，因为最初的三层架构的项目启动的话，最常见的我们需要配置一个容器，也就是一个服务器的环境，但是 Spring Boot 不需要，因为 Spring Boot 内置了 Tomcat 的 starter。

当使用 Spring Boot 原生的内置 tomcat 作为服务器的时候，效果如下：

更改配置前的运行结果：

上两个图可以看出，Spring Boot中默认的是Tomcat服务器，端口为8080，context path为/，现在更改端口号为8081，context path为/chan，修改如下：

server:
  port: 8081 #修改服务器的端口号
  servlet:
    context-path: /chan #修改服务器的端口号
  tomcat:
    uri-encoding: UTF-8 #配置Tomcat的URL编码

2. 属性注入

在Spring Boot中，配置文件有两种不同的格式，一种是 properties ，另一种是 yml 。Spring boot 项目中同时存在 application.properties 和 application.yml 文件时，两个文件都有效，但是 application.properties 的优先级会比 application.yml 高。 为什么这么说呢？ 首先，我在 application.properties 文件中配置了：

server.port=8085

 然后在 application.yml 文件中配置了：

server:
  port: 8086

启动项目，控制台输出：

[  main] s.b.c.e.t.TomcatEmbeddedServletContainer : Tomcat started on port(s): 8085 (http)

虽然 properties 文件比较常见，但是相对于 properties 而言， yml 更加简洁明了。另外， yml 中的数据是有序的，这点不同于 properties ，在一些需要路径匹配的配置中，顺序就显得尤为重要了，此时我们都会采用yml。

(1)关于properties

①位置问题

在我们创建了 Spring Boot 的项目之后，默认的 resources 目录下就会有一个 application.properties 文件，可以在其中进行配置，但是这个文件并非唯一的配置文件。在 Spring Boot 中，一共有4个存放 application.properties 的地方。

如图，这四个位置是默认位置，即 Spring Boot 启动的时候，默认会从这四个位置按顺序去查找相关属性并加载。

②普通的属性注入

Spring Boot 源自于 Spring ，所以 Spring 中存在的属性注入，在 Spring Boot 中也一样的存在。由于 Spring Boot 中默认会自动的加载 application.properties 文件，因此简单的属性注入可以直接写在这个配置文件中。

定义Book类：

public class Book {
    private Long id;
    private String name;
    private String author;
    //省略 getter/setter
}

然后在 application.properties 文件中定义属性：

book.name=三国演义
book.author=罗贯中
book.id=1

按照传统的方式（ Spring 中的方式），可以直接通过 @Value 注解将这些属性注入到 Book 对象中：

@Component
public class Book {
    @Value("${book.id}")
    private Long id;
    @Value("${book.name}")
    private String name;
    @Value("${book.author}")
    private String author;
    //省略getter/setter
}

注意：

Book 对象本身也要交给 Spring 容器去管理，如果 Book 没有交给 Spring 容器，那么 Book 中的属性也无法从 Spring 容器中获取到值。

配置完成后，在 Controller 或者单元测试中注入 Book 对象，启动项目，就可以看到属性已经注入到对象中了。

一般来说，我们在 application.properties 文件中主要存放系统配置，这种自定义配置不建议放在该文件中，可以自定义 properties 文件来存在自定义配置。

例如在 resources 目录下，自定义 book.properties 文件，内容如下：

book.name=三国演义
book.author=罗贯中
book.id=1

此时，项目启动并不会自动的加载该配置文件，如果是在xml配置中，可以通过如下的方式引用 properties 文件：

<context:property-placeholder location="classpath:book.properties"/>

如果是想在 Spring Boot 这样的没有 xml 的工程中，使用Java配置的话，可以使用 @PropertySource 来引入配置：

@Component
@PropertySource("classpath:book.properties")//本注解yml不支持
public class Book {
    @Value("${book.id}")
    private Long id;
    @Value("${book.name}")
    private String name;
    @Value("${book.author}")
    private String author;
    //getter/setter
}

这样，当项目启动时，就会自动加载 book.properties 文件。

说白了，多了个 @PropertySource 注解，就将这些普通的属性注入从 application.properties 牵引到 xxx.properties 了。

③类型安全的属性注入

在 Spring Boot 中引入了类型安全的属性注入，如果采用 Spring 中的配置方式，当需要配置的属性较多的时候，工作量就非常的大了，并且容易出错。

使用类型安全的属性注入，可以有效的解决这个问题。

@Component
@PropertySource("classpath:book.properties")//yml中不支持
@ConfigurationProperties(prefix = "book")//代替了上面的@Value们
public class Book {
    private Long id;
    private String name;
    private String author;
    //省略getter/setter
}

这里，在原来的基础上，主要是引入了 @ConfigurationProperties(prefix = “book”) 注解，并且在注解中配置了属性的前缀，此时会自动的将 Spring 容器中的对应的数据注入到对象对应的属性中，就不用 @Value 注解挨个的注入了，减少了工作量，并且避免出错。

(2)关于yml

①位置问题

首先 application.yml 在 Spring Boot 中可以写在四个不同的位置，分别是如下位置：

  1. 项目根目录下的 config 目录中
  2. 项目根目录下
  3. classpath 下的 config 目录中
  4. classpath 目录下

四个位置中的 application.yml 文件的优先级按照上面列出的顺序依次降低。即如果有同一个属性在四个文件中都出现了，以优先级高的为准。

②数组注入

除了刚刚的那些， yml 也支持数组注入。application.yml：

my:
  servers:
	- dev.example.com
	- another.example.com

这段数据可以绑定到一个带 Bean 的数组中：

@ConfigurationProperties(prefix="my")//刚说的安全的属性注入的方式
@Component
public class Config {

	private List<String> servers = new ArrayList<String>();

	//getter&setter省略...
}

项目启动后，配置中的数组会自动存储到 servers 集合中。当然， yml 不仅可以存储这种简单的数据，也可以在集合中存储对象。例如下面这种，application.yml：

redis:
  redisConfigs:
    - host: 192.168.66.128
      port: 6379
    - host: 192.168.66.129
      port: 6380

将 host 和 port 作为属性构成一个 SingleRedisConfig 对象之后，这个可以被注入到如下的类中：

@Component
@ConfigurationProperties(prefix = "redis")
public class RedisCluster {
    private List<SingleRedisConfig> redisConfigs;
	//省略getter/setter
}

(3)优缺点

不同于 properties 文件的无序， yml 配置是有序的，这一点在有些配置中是非常有用的，例如在 Spring Cloud Zuul 的配置中，当我们配置代理规则时，顺序就显得尤为重要了。当然 yml 配置也不是万能的，例如，yml 配置目前不支持 @PropertySource 注解，因此自定义的 yml 目前暂时不支持使用注解直接注入到 Spring Boot 项目中，所以不能像 properties 一样自定义成 book.yml ，只能写在 application.yml 文件中。

3. profile

当需要开发环境和生产环境和上线环境切换的时候，因为每个环境的某些配置可能不相同，举个最常见的例子就是数据库的地址，那这个时候为了方便开发，我们会需要新建配置文件的分支。然后将统一的配置写到主要的配置文件里面，将可能有变化的配置放入新建的配置文件中。

例如：

主：application.yml

dev：application-dev.yml

pro：application-pro.yml

然后在主 application.yml 中添加 spring.profiles.active=dev/pro ，来指定使用哪一个从配置文件中的配置。

另外有一个小技巧：当在 pom.xml 中添加依赖的时候，有一个小窍门，如果记不住的话可以记一下：

Spring Boot 的的话就是 spring-boot-starter-xxx ，其它的的话就是 xxx-spring-boot-starter 。适合大多数的情况。
