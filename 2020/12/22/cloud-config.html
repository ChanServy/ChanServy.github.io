Spring Cloud Config

一、简介

一个分布式系统配置管理的解决方案，包含了Client和Server。配置文件放在Server端，通过接口的形式提供给Client。

Spring Cloud Config的主要功能：

  • 集中管理各个环境、各个微服务的配置文件
  • 提供服务端和客户端支持
  • 配置文件修改后，可以快速生效
  • 配置文件通过git、svn进行管理，天然支持版本回退功能。
  • 支持高并发查询，也支持多种语言开发。

创建成功后，启动类添加注解，开启config server功能：

package ink.csy.server;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.cloud.config.server.EnableConfigServer;

/**
 * @author chan
 * @version V1.0
 * @Package ink.csy.server
 * @date 2020/10/17 下午10:06
 * @Description
 */
@SpringBootApplication
@EnableConfigServer
public class ConfigServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(ConfigServerApplication.class,args);
    }
}

然后在配置文件中配置仓库的基本信息

spring.application.name=config-server
server.port=8081
# 配置文件仓库地址
spring.cloud.config.server.git.uri=https://github.com/ChanServy/cloudConfigRepo.git
# 仓库中,配置文件的目录,每个目录就分别是这个父工程中的每个微服务的配置文件,目录名就是微服务名
spring.cloud.config.server.git.search-paths=client01
# 仓库的用户名密码
spring.cloud.config.server.git.username=ChanServy
spring.cloud.config.server.git.password=

启动项目之后，就可以配置文件了。访问地址：http://localhost:8081/client01/prod/master

实际上，访问地址有如下规则：

  • /{application}/{profile}/{label} http://localhost:8081/client01/prod/master
  • /{application}-{profile}.yml http://localhost:8081/client01-prod.yml
  • /{application}-{profile}.properties http://localhost:8081/client01-prod.properties
  • /{label}/{application}-{profile}.yml http://localhost:8081/master/client01-prod.yml
  • /{label}/{application}-{profile}.properties http://localhost:8081/master/client01-prod.properties

application表示配置文件名，也就是每个configclient的配置文件中配置的微服务名

profile表示配置文件的profile

label表示git的分支，参数可选，默认是master

接下来，修改配置文件的话并且重新提交到github，此时，刷新configserver的接口。就可以及时的看到更改。

四、配置

前面在config-server中将目录client01写死了，只是做测试用，但是开发的时候不可能将这块写死，我们可以使用占位符灵活的控制查询目录。

修改config-server的配置文件

spring.application.name=config-server
server.port=8081
# 配置文件仓库地址
spring.cloud.config.server.git.uri=https://github.com/ChanServy/cloudConfigRepo.git
# 仓库中,配置文件的目录,每个目录就分别是这个父工程中的每个微服务的配置文件,目录名就是微服务名
#spring.cloud.config.server.git.search-paths=client01
spring.cloud.config.server.git.search-paths={application}

这里的{application}占位符，表示连接上来的是client01这个微服务的spring.application.name属性的值。

在没配置这块之前，client01这个微服务的spring.application.name属性的值只在配置文件名的“-”前面起作用，配置之后，client01这个微服务的spring.application.name属性的值在目录也起了作用。

在config-server中，也可以用{profile}表示client的spring.cloud.config.profile，label同理。

五、安全管理

为了防止用户直接通过访问config-server看到配置文件内容，我们可以用spring security来保护config-server的接口。

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>

添加玩依赖后，config-server的接口就被自动保护起来了。

security默认自动生成的密码不好记，所以我们可以在config-server中自己配置用户名和密码，在config-server的配置文件中添加配置：

# spring security保护config server接口
spring.security.user.name=chan
spring.security.user.password=111

然后，为了config-client可以访问到加密后的config-server接口，我们在config-client的bootstrap.properties文件中添加如下配置

# 认证
spring.cloud.config.username=chan
spring.cloud.config.password=111

六、服务化

前面的配置都是直接在config-client中写死config-server的地址。这样不好管理，我们将他们注册到eureka注册中心中。首先启动eureka，然后在config-server和config-client中添加eureka的依赖：

<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-client</artifactId>
</dependency>

然后在application.properties中添加注册信息

# 将服务注册到eureka
eureka.client.serviceUrl.defaultZone=http://localhost:20000/eureka/

然后修改config-client的配置文件，不再写死config-server的地址了：

# 下面三行配置,分别对应 config-server 中的 {application},{profile}以及{label}占位符
spring.application.name=client01
spring.cloud.config.profile=dev
spring.cloud.config.label=master

# spring.cloud.config.uri=http://localhost:8081
# 开启通过eureka获取config-server的功能,不用在像上面将config-server的地址写死了
spring.cloud.config.discovery.enabled=true
# 配置config-server的服务名称,使config-client通过eureka找到配置中心
spring.cloud.config.discovery.service-id=config-server

server.port=8082

# 认证
spring.cloud.config.username=chan
spring.cloud.config.password=111

# 注册到eureka
eureka.client.serviceUrl.defaultZone=http://localhost:20000/eureka/

七、动态刷新

当配置文件发生了改变并提交到github后，config-server可以及时的感知变化，但是config-client不会，默认情况下，config-client只有重启才能加载到最新的配置文件。这当然违背初衷。

我们要实现动态刷新，首先给config-client添加如下依赖：

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>

然后添加配置，是refresh端点暴露：

# 自动刷新端点
management.endpoints.web.exposure.include=refresh

最后，在给config-client中使用了配置文件的地方加上@RefreshScope注解，这样，当配置改变后，只需要调用refresh端点，config-client中的配置就可以自动刷新。

/**
 * @author chan
 * @version V1.0
 * @Package ink.csy.client.controller
 * @date 2020/10/17 下午11:14
 * @Description
 */
@RestController
@RefreshScope
public class TestController {
    @Value("${chan}")//编写bootstrap配置文件后就表示链接到配置中心
    String name;
    @GetMapping("/hello")
    public String hello(){
        return name;
    }
}

重启config-client之后，只要配置文件再次发生变化，提交到github之后，我们直接用postman发送POST请求，调用 http://localhost:8082/actuator/refresh 接口即可，配置文件就会自动刷新。注意，这里的动态刷新是站在config-client层面的，而非config-server层面，站在配置中心层面的话需要用到消息的广播机制。

八、请求失败重试

config-client在调用config-server时，一样也可能发生请求失败的问题，这个时候，我们可以配置一个请求失败后重试的功能。要给config-client添加重试功能，只需要添加如下的依赖：

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.retry</groupId>
    <artifactId>spring-retry</artifactId>
</dependency>

然后，修改client的配置文件：

# 开启失败快速响应,因为不开的话,如果client启动加载不到配置中心的配置文件是不会立刻响应的,会等到接口调用的时候,那就没意义了
spring.cloud.config.fail-fast=true

然后，我们为了测试，注释掉client中链接配置中心的用户名和密码，重启config-client，此时加载配置文件失败，就会自动重试。也可以通过如下的配置保证服务的可用性：

# 开启失败快速响应,因为不开的话,如果client启动加载不到配置中心的配置文件是不会立刻响应的,会等到接口调用的时候,那就没意义了
spring.cloud.config.fail-fast=true
# 其他配置保证服务可用性
# 请求重试的初始间隔时间
spring.cloud.config.retry.initial-interval=1000
# 最大重试次数
spring.cloud.config.retry.max-attempts=6
# 重试时间间隔乘数
spring.cloud.config.retry.multiplier=1.1
# 最大间隔时间
spring.cloud.config.retry.max-interval=2000
